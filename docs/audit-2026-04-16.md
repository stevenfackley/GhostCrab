### H1: `ConfigRepositoryImpl.lastEtag` is not thread-safe — data race under concurrent calls

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/data/impl/ConfigRepositoryImpl.kt`, lines 19, 31

`lastEtag` is a `var` field mutated by `getConfig()` and read by `updateConfig()`. `ConfigRepositoryImpl` is a Koin `single` (shared across coroutines). Two concurrent `getConfig()` calls can interleave with `updateConfig()`, producing a stale or wrong ETag.

The field is not declared `@Volatile`, not protected by a mutex, and is not a `StateFlow`/`AtomicReference`. The `GatewayConnectionManagerImpl` wraps its own mutation behind a `Mutex`, but `ConfigRepositoryImpl` has no such protection.

**Fix:** Protect with the same `Mutex` used in the manager, or use `AtomicReference<String?>`:

```kotlin
private val lastEtag = AtomicReference<String?>(null)

override suspend fun getConfig(): OpenClawConfig {
    val client = connectionManager.requireClient()
    val (sections, etag) = client.getConfig()
    lastEtag.set(etag)
    return OpenClawConfig(sections = sections, etag = etag)
}

override suspend fun updateConfig(section: String, value: JsonElement) {
    connectionManager.requireClient().updateConfig(section, value, lastEtag.get())
}
```

### H2: `ConnectionProfileStore.openEncryptedPrefs()` creates a new `MasterKey` and `EncryptedSharedPreferences` instance on every call — performance and correctness risk

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/data/storage/ConnectionProfileStore.kt`, lines 100–111

`openEncryptedPrefs()` is called from `saveToken()`, `getToken()`, and `clearToken()`. Each call runs `MasterKey.Builder(...).build()` which is a synchronous Keystore operation (can block 20–200ms) and then `EncryptedSharedPreferences.create()`. This is:

1. Slow — each token read/write pays the Keystore initialization cost.
2. Potentially unsafe — `EncryptedSharedPreferences` documentation warns against creating multiple instances pointing at the same file simultaneously; this can trigger internal locking issues.
3. The call to `clearToken()` inside the `catch` block in `getToken()` calls `openEncryptedPrefs()` again inside a catch, which can itself throw, masking the original failure.

**Fix:** Lazily initialize and cache a single instance, protected for thread safety:

```kotlin
private val encryptedPrefs: android.content.SharedPreferences by lazy {
    val masterKey = MasterKey.Builder(context)
        .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
        .build()
    EncryptedSharedPreferences.create(
        context, "ghostcrab_tokens", masterKey,
        EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
        EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM,
    )
}
```

Note: `lazy` in Kotlin uses `LazyThreadSafetyMode.SYNCHRONIZED` by default, which is correct here.

### H3: `GatewayConnectionManagerImpl` has a `settingsRepository: SettingsRepository?` nullable parameter that silently defaults security off

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/data/impl/GatewayConnectionManagerImpl.kt`, line 30

```kotlin
private val settingsRepository: SettingsRepository? = null,
```

When `settingsRepository` is `null` (e.g. in tests that don't supply it), `allowCleartext` defaults to `false`. But the **intent** of the default is to deny cleartext. The problem is the default `null` is invisible — any caller could accidentally instantiate this without a settings repo and not realise the security check is inactive (rather than fail-secure). Tests that construct `GatewayConnectionManagerImpl(factory)` without a settings repo are relying on the `null` default accidentally being fail-safe, which is fragile.

More critically: `DefaultClientFactory` is used as the default factory parameter too (`= DefaultClientFactory`). This means a no-arg constructor could build a live implementation with no settings repo. If a future refactor passes a permissive interceptor, the security default would silently flip.

**Fix:** Make `settingsRepository` non-nullable. Where tests don't need it, provide a trivial stub:

```kotlin
class GatewayConnectionManagerImpl(
    private val clientFactory: OpenClawApiClientFactory = DefaultClientFactory,
    private val settingsRepository: SettingsRepository,
) : GatewayConnectionManager
```

Update `DataModule` accordingly (it already passes `get()` so no change needed there). Update tests to pass a `FakeSettingsRepository(false)`.

### H4: `NsdDiscoveryServiceImpl.activeChannel` is accessed from both the `callbackFlow` lambda and `stopDiscovery()` without synchronization — race condition

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/data/discovery/NsdDiscoveryServiceImpl.kt`, lines 45, 55, 100–101, 111–113

`activeChannel` is a `var` field on the class (single instance, Koin `single`). It is written in `startDiscovery()` on the flow's coroutine and read/nulled in `stopDiscovery()` which may be called from a different coroutine. There is no `@Volatile`, no mutex.

The `resolveMutex` protects NSD resolution correctly, but nothing protects `activeChannel`. If `stopDiscovery()` reads `activeChannel` concurrently with the `awaitClose` block writing `null` to it, you get a race.

**Fix:** Mark `activeChannel` as `@Volatile`, or use `AtomicReference<SendChannel<DiscoveredGateway>?>`.

### H5: `safeRequest` swallows `ConfigValidationException` under the final `catch (e: Exception)` — exception type already thrown from `updateConfig` could be re-wrapped as `GatewayUnreachableException`

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/data/api/OpenClawApiClient.kt`, lines 288–309

`safeRequest` has explicit re-throw guards for `GatewayAuthException` and `GatewayApiException`, but **not** for `ConfigValidationException`. The `updateConfig` method throws `ConfigValidationException` inside `safeRequest`. If anything in the path between the throw site and `safeRequest`'s catch re-throws (e.g., a Ktor deserialization path), `ConfigValidationException` would fall through to the final `catch (e: Exception)` and be re-wrapped as `GatewayUnreachableException`. This would break the 422-handling logic in `ConfigEditorViewModel.confirmSave()`.

Current guard chain is:
```
CancellationException → rethrow
GatewayAuthException → rethrow
GatewayApiException → rethrow
```

`ConfigValidationException` is a `GatewayException` subclass, not a `GatewayApiException` subclass, so it is not caught by the `GatewayApiException` guard. It would fall to `catch (e: Exception) → GatewayUnreachableException`.

**Fix:** Add a guard:
```kotlin
} catch (e: GatewayException) {
    throw e  // covers ConfigValidationException, GatewayTlsException, etc.
```
Place this before the `HttpRequestTimeoutException` handler to protect all `GatewayException` subclasses.

### I3: `DashboardViewModel` health poller reads `_state.value` via unsafe smart cast

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/ui/dashboard/DashboardViewModel.kt`, lines 84–98

```kotlin
(_state.value as? DashboardUiState.Ready)?.let { current ->
    _state.value = current.copy(...)
}
```

This pattern reads `_state.value`, enters the `let`, then writes `_state.value`. If another coroutine modifies `_state` between the read and the write (e.g., `disconnect()` calls `connectionManager.disconnect()` which emits `Disconnected`, causing the `connectionState.collect` lambda to set `_state.value = DashboardUiState.Disconnected`), the `let` executes on a stale snapshot and overwrites `Disconnected` with a `Ready` copy.

The correct approach is an atomic `MutableStateFlow.update {}`:

```kotlin
_state.update { current ->
    if (current is DashboardUiState.Ready) current.copy(...) else current
}
```

This applies to lines 84, 93, and all similar `(_state.value as? ...).let` patterns in `ConfigEditorViewModel`, `ModelManagerViewModel`, `AIRecommendationViewModel`, and `SettingsViewModel`.

### I4: `DashboardViewModel` health poll delay comes **after** the health check, not before — first check happens with no delay after models load, but subsequent cycle is delay-then-check, creating asymmetric timing

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/ui/dashboard/DashboardViewModel.kt`, lines 80–107

The loop structure is:
```
while (isActive) {
    try { healthChecker() } catch { ... }
    delay(POLL_INTERVAL_MS)   // delay at END
}
```

The first poll runs immediately (correct — acceptable for "first check fast"). However, a `Degraded` transition on two failures will happen at t=0 (first fail) and t=30s (second fail). If the intent is "two consecutive failures within a polling window", this is fine. But if someone reads the KDoc ("two consecutive poll failures") and expects the first failure to start a timer, the second failure at exactly 30s may feel slow.

More importantly: if `startDashboard` is called while the previous `pollJob` is still in `delay()`, the old delay resumes after the new job starts (the old job is cancelled by `scanJob?.cancel()` in `connectionState.collect`). This is correct. But the lack of a leading delay means reconnect → immediate health check occurs, which is actually good. This is Low priority but the asymmetry is worth documenting.

No code change strictly required, but add a comment explaining the intentional asymmetry.

### I5: `OnboardingRepositoryImpl` serializes `OnboardingStep` via a hand-rolled `when`/`name` extension and a separate `stepFromName` function — fragile, not exhaustive by compiler

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/data/impl/OnboardingRepositoryImpl.kt`, lines 67–86

```kotlin
private val OnboardingStep.name: String
    get() = when (this) { ... }
```

`OnboardingStep` is a sealed interface with `data object` leaves. The `when` in `name` is exhaustive (compiler would catch a missing arm). But `stepFromName` is a plain `when` on `String` with an `else -> OnboardingStep.Welcome` fallback — if a new step is added to the sealed interface, `stepFromName` will silently fall back to `Welcome` for the new step name, hiding the bug.

**Fix:** Since `OnboardingStep` is a sealed interface with `data object` leaves, use `ordinal` encoding via a companion `values()` list, or annotate the `stepFromName` fallback to make the risk visible:

```kotlin
// Alternative: use name property from the data object class name
private fun stepFromName(name: String): OnboardingStep =
    OnboardingStep::class.sealedSubclasses
        .mapNotNull { it.objectInstance }
        .firstOrNull { it::class.simpleName == name }
        ?: OnboardingStep.Welcome
```

Or more simply, document the `else` branch with a `// NEW STEPS: update this when` comment. The current code has no such note, which is a trap.

### I6: `AIRecommendationViewModel.applySelectedChanges` applies changes sequentially with no atomicity — partial failure leaves gateway in inconsistent state

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/ui/airecommend/AIRecommendationViewModel.kt`, lines 139–155

```kotlin
snapshot.selectedChanges.forEach { change ->
    configRepository.updateConfig(change.section, patch)
}
```

If the second `updateConfig` fails, the first has already been committed to the gateway. The UI surfaces `applyError` but does not indicate which changes succeeded. The user has no way to know the gateway is in a partial state.

This is not fixable at the client level without a gateway transaction API, but the error message should be explicit:

```kotlin
val results = snapshot.selectedChanges.mapIndexed { i, change ->
    runCatching { configRepository.updateConfig(change.section, patch) }
        .also { if (it.isFailure) return@mapIndexed Pair(change, it) }
    Pair(change, Result.success(Unit))
}
val failures = results.filter { it.second.isFailure }
if (failures.isNotEmpty()) {
    // surface: "Applied X of Y changes. Failed: [section.key, ...]"
}
```

The current error message just says `e.message ?: "Failed to apply changes"` with no count or identity of what succeeded.

### I7: `UiModule` creates an `OpenClawApiClient` directly inside a `viewModel { }` lambda — data layer instantiation in DI UI module

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/di/UiModule.kt`, lines 23–27

```kotlin
viewModel {
    DashboardViewModel(
        connectionManager = get(),
        modelRepository = get(),
        healthChecker = { url ->
            val client = OpenClawApiClient.unauthenticated(url)
            try { client.health() } finally { client.close() }
        },

}
```

`OpenClawApiClient` is a data-layer class. Instantiating it inside `UiModule` violates the layer boundary: UI module now has a direct compile-time dependency on a data-layer implementation class. If `OpenClawApiClient`'s constructor changes, the UI module breaks.

The `healthChecker` lambda is a function type `suspend (String) -> HealthResponse`, which is correct for testability. The issue is the *module location* of the wiring. Move the lambda definition to `DataModule`:

```kotlin
// DataModule.kt
single<suspend (String) -> HealthResponse> { url ->
    val client = OpenClawApiClient.unauthenticated(url)
    try { client.health() } finally { client.close() }
}
```

Or define a named `HealthChecker` typealias and bind it in `DataModule`.

### I8: `DashboardUiState.HealthSnapshot.isStale` calls `System.currentTimeMillis()` inside a `val` getter — non-deterministic, untestable, causes spurious recomposition

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/ui/dashboard/DashboardUiState.kt`, lines 20–22

```kotlin
val isStale: Boolean
    get() = lastOkMs != null && (System.currentTimeMillis() - lastOkMs > STALE_THRESHOLD_MS)
```

This is a computed property that returns different values each time it is called — the same `HealthSnapshot` instance will return `isStale = false` then `isStale = true` as time passes, without any state update. Compose stability analysis will mark `HealthSnapshot` as unstable (it has a property whose value changes externally), which triggers extra recompositions.

More importantly: the `HealthCard` composable reads `health.isStale` during composition, but there is no mechanism to re-trigger composition when the threshold is crossed (no `LaunchedEffect`, no coroutine). The stale indicator will only update when the parent recomposes for another reason (e.g., a health poll result).

**Fix:** Compute `isStale` in the ViewModel at poll time and store it in the state, or use a `derivedStateOf` with a timer-driven state update at the UI layer. At minimum, remove the live clock from the data class:

```kotlin
data class HealthSnapshot(
    val lastOkMs: Long?,
    val lastError: String?,
    val isStale: Boolean,  // set by ViewModel at emission time
)
```

### I9: `QrScanScreen` has two `LaunchedEffect(Unit)` blocks that both run on first composition — the order is unspecified and one could cancel the other

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/ui/connection/QrScanScreen.kt`, lines 94–117

```kotlin
LaunchedEffect(Unit) {   // permission check
    ...
}

LaunchedEffect(Unit) {   // events collection
    viewModel.events.collect { ... }
}
```

Two `LaunchedEffect(Unit)` blocks are two separate coroutine launchers keyed on the same key. This is legal in Compose — they run as independent coroutines in the same `CoroutineScope`. However:

1. The events-collection `LaunchedEffect` calls `onNavigateToManualEntry(event.url)` which triggers navigation, potentially destroying the composition before the permission `LaunchedEffect` completes its side effects.
2. The permission `LaunchedEffect` and the `LaunchedEffect(uiState)` (line 110) both touch `viewModel.onPermissionGranted()` / `viewModel.onScanAgain()`. If `uiState` changes to `InvalidQr` before the events collector fires `NavigateToManualEntry`, both effects run concurrently and the snackbar might try to show on a navigated-away screen.

**Fix:** Use distinct keys:
```kotlin
LaunchedEffect("permission") { ... }
LaunchedEffect("events") { ... }
```

And guard the `InvalidQr` LaunchedEffect against acting after navigation:
```kotlin
LaunchedEffect(uiState) {
    if (uiState !is QrScanUiState.InvalidQr) return@LaunchedEffect
    ...
}
```

### I10: `ManualEntryViewModel.connect()` reads `connectionManager.connectionState.value` after `connect()` returns — TOCTOU on connection state

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/ui/connection/ManualEntryViewModel.kt`, lines 81–89

```kotlin
connectionManager.connect(state.url.trim(), token)
}.onSuccess {
    val connected = connectionManager.connectionState.value
    val profile = ConnectionProfile(
        ...
        displayName = (connected as? GatewayConnection.Connected)?.displayName ?: state.url.trim(),
```

`connect()` is `suspend` and when it returns on success, `connectionState` should be `Connected`. But this is an implicit assumption. If another coroutine (e.g., a fast background health failure) races to set `connectionState` to `Error` or `Disconnected` between `connect()` returning and `connectionManager.connectionState.value` being read, the `displayName` falls back to the raw URL.

This is low probability in practice but is a TOCTOU by design. The `connect()` method's contract (KDoc says it emits `Connected` on success, Error+rethrows on failure) should be used directly: if `connect()` returned without throwing, capture the state immediately in the `connect()` call's scope or use the return value.

**Fix:** Return the display name from `connect()` in the contract — but since that contract is frozen, the simpler fix is to read state atomically:

```kotlin
val connectedState = connectionManager.connectionState.value as? GatewayConnection.Connected
val displayName = connectedState?.displayName ?: state.url.trim()
```

This is what the code already does, but it should be documented as "the connect() return guarantees Connected state; this read is safe".

### L5: `PrivacySafeUncaughtExceptionHandler` logs the sanitized stack trace to `Log.e()` BEFORE forwarding to `delegate`

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/crash/PrivacySafeUncaughtExceptionHandler.kt`, lines 21–23

The `delegate` (system handler) causes the process to terminate. `Log.e()` before delegate is fine on Android — the log should flush before crash. ### L6: `NsdDiscoveryServiceImpl` acquires a `MulticastLock` with `setReferenceCounted(true)` but releases it without checking if it was acquired

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/data/discovery/NsdDiscoveryServiceImpl.kt`, lines 50–53, 107

```kotlin
val multicastLock = wifiManager.createMulticastLock("ghostcrab_mdns").apply {
    setReferenceCounted(true)
    acquire()
}
// ...
if (multicastLock.isHeld) multicastLock.release()
```

`isHeld` check before `release()` is correct. `setReferenceCounted(true)` means `acquire()`/`release()` calls must be balanced. Since `acquire()` is called exactly once and `release()` is guarded by `isHeld`, this is balanced — good. However, if `acquire()` throws (e.g., `WifiManager` service not available), the flow will throw before `awaitClose` is reached, leaving the lock in an undefined state. Wrap the acquire:

```kotlin
runCatching { multicastLock.acquire() }
```

And only release if actually acquired.

### L7: `ScanViewModel.onCleared()` does not call `super.onCleared()` and does not call `discoveryService.stopDiscovery()`

**File:** `app/src/main/kotlin/com/openclaw/ghostcrab/ui/connection/ScanViewModel.kt`, lines 175–177

```kotlin
override fun onCleared() {
    scanJob?.cancel()
}
```

`super.onCleared()` is not called. For `ViewModel`, `onCleared()` in the base class calls `viewModelScope.cancel()`. In practice this is fine because `viewModelScope` is already canceled when `onCleared()` is called by the framework, but omitting `super` is a hygiene issue.

More importantly: `discoveryService.stopDiscovery()` is not called in `onCleared()`. When the screen is popped from the back stack, NSD discovery may continue running until the `callbackFlow` is garbage-collected. The multicast lock would not be released. This is a real resource leak.

**Fix:**

```kotlin
override fun onCleared() {
    super.onCleared()
    scanJob?.cancel()
    // viewModelScope is cancelled so we can't launch; use a raw coroutine here
    // or call stopDiscovery synchronously via runBlocking (acceptable in onCleared)
}
```

Since `stopDiscovery()` is `suspend`, the cleanest fix is to ensure the `callbackFlow`'s `awaitClose` block handles cleanup (it does, via `nsdManager.stopServiceDiscovery`), and rely on `scanJob?.cancel()` propagating cancellation to the `callbackFlow`. Confirm the `callbackFlow` is actually cancelled when `scanJob` is cancelled — it is, because `collect` is inside `scanJob`'s scope. So the existing code is functionally correct but `super.onCleared()` should be added.

### T1: No test for `ConfigEditorViewModel.confirmSave` with 412 concurrent-edit — the most complex error path in the VM

**File:** `app/src/test/kotlin/com/openclaw/ghostcrab/ui/ConfigEditorViewModelTest.kt` (presumed)

The 412 conflict path in `ConfigEditorViewModel.confirmSave()` involves: (a) calling `updateConfig` → 412 → (b) calling `getConfig()` to reload → if reload succeeds, set `concurrentEditSection` → if reload also fails, set `Error`. This is a two-step failure path with a nested `try/catch`. No test file was found for `ConfigEditorViewModelTest` — this entire ViewModel has no unit tests. Given the complexity of `confirmSave()`, this is a significant gap.

**Add tests for:**
- `confirmSave` → success → `saveSuccess = true`, `pendingChanges` cleared
- `confirmSave` → 412 → reload succeeds → `concurrentEditSection` set
- `confirmSave` → 412 → reload fails → `Error` state
- `confirmSave` → 422 → `ConfigValidationException` → `fieldErrors` updated

## What Is Done Well

The following aspects are notably high quality and worth preserving:

1. **`GatewayException` sealed hierarchy**: every leaf has `isRetryable`, `url` context is present where relevant, and message strings match the CLAUDE.md spec ("exact URL, HTTP status code, exception class"). `GatewayAuthException` and `GatewayApiException` are clearly separated.

2. **`safeRequest` in `OpenClawApiClient`**: correctly re-throws `CancellationException` first, then explicitly re-throws domain exceptions before the general mapping. This is the correct pattern and is often botched in practice.

3. **`NsdDiscoveryServiceImpl` resolve serialization**: the `resolveMutex` protecting NsdManager single-in-flight constraint is correct, and the `resolveWithRetry` fallback is sensible. `DEPRECATION` suppression with a comment explaining the minSdk constraint is proper.

4. **`SettingsViewModel.withReady` inline helper**: clean ergonomic pattern avoiding repeated `(_state.value as? SettingsUiState.Ready)?.let` boilerplate.

5. **`DashboardViewModel` health poll state machine**: `consecutiveFailures` counter with explicit `DEGRADED_FAILURE_THRESHOLD` constant, with `Ready`-before-first-poll guarantee documented and tested.

6. **`PrivacySafeUncaughtExceptionHandler`**: sanitizes bearer tokens and URL credentials from stack traces before logging. The regex handles case-insensitive `Bearer` and URL-encoded credentials.

7. **`ConnectionProfileStore` DataStore/EncryptedSharedPreferences split**: correct architectural decision to keep metadata in DataStore (observable Flow) and tokens in EncryptedSharedPreferences (no flow needed for a secret). The `ProfileNeedsReauthException` + clear pattern on decryption failure is production-correct.

8. **`ManualEntryViewModel` URL validation**: correctly checks scheme prefix before running `Patterns.WEB_URL`, then separately validates port range via URI parsing. Defensive and ordered correctly.

9. **Test fake strategy**: all ViewModelTests use hand-written fakes (not mocks) against the domain interfaces, which correctly exercises the ViewModel in isolation without leaking implementation details. Fakes are defined in the test file, not as shared fixtures — good for test isolation.

10. **`QrScanViewModel` has no dependencies** and doesn't need them — the camera permission and CameraX lifecycle wiring lives in the screen, keeping the ViewModel pure and easily testable.
